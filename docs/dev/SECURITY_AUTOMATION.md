# Automated security controls

Taxonomy treats security analysis as a set of independent, evidence-producing gates. A successful Maven build does not replace source analysis, dependency scanning, secret detection, configuration scanning, or supply-chain provenance checks.

## Control matrix

| Control | Scope | Blocking policy | Evidence |
|---|---|---|---|
| CodeQL Java/Kotlin | Production Java source and generated bytecode | SARIF `error` findings or security severity ≥ 7.0 | Raw SARIF plus `target/codeql-java-gate.json`, retained 90 days |
| CodeQL JavaScript/TypeScript | Browser JavaScript and TypeScript-compatible source | SARIF `error` findings or security severity ≥ 7.0 | Raw SARIF plus `target/codeql-javascript-gate.json`, retained 90 days |
| Trivy vulnerability scan | Repository and packaged application dependencies | HIGH or CRITICAL findings | SARIF uploaded to code scanning and retained as an artifact |
| Trivy secret scan | Entire checked-out repository | HIGH or CRITICAL findings | Same SARIF evidence |
| Trivy misconfiguration scan | Workflows, Docker, Compose, and repository configuration | HIGH or CRITICAL findings | Same SARIF evidence |
| GitHub Action provenance | Every external `uses:` reference in `.github/workflows` | Reference is not an exact 40-character commit SHA | `target/supply-chain-pins.json` |
| Production image provenance | Every `FROM` image and active `image:` in production Compose | Image is not pinned to a SHA-256 manifest digest | `target/supply-chain-pins.json` |

## Pull-request coverage

The security workflows run for Java-only, JavaScript-only, Maven, Docker, Compose, workflow, and security-script changes. Source changes therefore cannot bypass analysis merely because no dependency descriptor changed.

CodeQL and Trivy remain separate jobs because they answer different questions:

- CodeQL analyses source and data-flow behavior.
- Trivy analyses dependency vulnerabilities, secrets, and configuration.
- The pinning gate verifies build-input immutability.

A pass in one control does not suppress or waive another control.

## Least privilege

Workflows default to `contents: read`. Additional permissions are declared only for the job that needs them, such as package publication, report publication, or SARIF upload. Checkout credentials are not persisted in ordinary build and analysis jobs.

## Updating pinned inputs

1. Resolve the desired release tag to its immutable Git commit or image manifest digest.
2. Replace the existing SHA or digest while retaining the human-readable release comment or image tag.
3. Review the upstream release notes and diff before accepting the update.
4. Run:

   ```bash
   python3 .github/scripts/check-supply-chain-pins.py
   ```

5. Verify CodeQL, Trivy, Maven, database compatibility, and container restart checks.
6. Commit the generated JSON evidence only as a CI artifact, not as repository state.

Mutable tags such as `@v7`, `:latest`, or an unqualified model `main` branch are not acceptable production inputs.

## Exceptions

There is no silent allow-list. A control exception requires a dedicated issue, an explicit risk owner, a bounded expiry date, and a code change that makes the exception visible to reviewers. Disabling a gate or adding `continue-on-error` is not an accepted remediation for a finding.
